# MIDHUB — Workflow Rules (обязательные правила работы)

> MIDHUB — **новый отдельный web-first проект**. Vector и Ubuntu используются
> ТОЛЬКО как архитектурные и технические референсы. Прямое копирование кода,
> токенов, стилей и компонентов из них **запрещено** без адаптации под MIDHUB.

---

## 0. Контекст проектов в этой папке

| Проект | Расположение | Стек | Роль для MIDHUB |
|---|---|---|---|
| **Vector** | корень `source/` (`src/`) | Next.js 16 + React 19 + Tailwind v4 + Prisma | Архитектурный референс (App Router, ui-kit, style_guide, i18n, auth) |
| **Ubuntu DS** | `Ubuntu Design System/` | Figma-экспорт + CSS токены | Референс DS-документации и token-структуры |
| **Ubuntu App** | `ubuntu-app/` | Vite + React + TS | Референс layered-DS подхода, tokens.css, router |
| **MIDHUB** | *(ещё не создан)* | *(уточняется)* | Новый проект |

**Web-first.** Все layout-решения принимаются под web (desktop → tablet → mobile),
а не под Telegram mini-app / mobile, как в Vector и Ubuntu.

---

## 1. Поиск перед созданием компонента

Перед созданием ЛЮБОГО нового компонента искать в таком порядке:

1. **MIDHUB DS** — `src/components/ds/`
2. **Текущие компоненты MIDHUB** — `src/components/`
3. **Vector** — `src/components/ui-kit/`, `src/components/style_guide/` (как паттерн)
4. **Ubuntu** — `ubuntu-app/src/components/`, `Ubuntu Design System/ui_kits/` (как паттерн)

Референс из Vector/Ubuntu даёт **архитектуру и API компонента** (props, variants,
states), но **не визуал** — визуал всегда из MIDHUB tokens.

### 1.1. Каждая новая ссылка из Figma — сперва аудит DS (обязательно)

Для **любой** новой ссылки/ноды из Figma, прежде чем верстать:

1. Получить структуру ноды (`get_metadata`) и разобрать её на части
   (атомы и под-блоки), а не как один монолит.
2. По каждой части пройтись по DS (`src/components/ds/` и `ds/composite/`,
   `src/components/ds/index.ts`) и найти готовый компонент:
   кнопка → `Button`, инпут → `Input`, чекбокс → `Checkbox`, ссылка → `Link`,
   удаление → `DeleteButton`, строка списка → `Item`, бейдж/статус → `Badge`,
   шапка таблицы → `TableHeader`, аккордеон-вопрос → `QuestionCard`,
   поиск → `SearchBar`, панель действий → `Footer` и т.д.
3. Верстать с нуля **только** то, чего в DS реально нет (после проверки).
4. В ответе **явно перечислить**: какие DS-компоненты переиспользованы и что
   действительно новое (с обоснованием, почему его не было).

Цель — не верстать лишнее и не плодить дубли: большинство фич — это композиция
уже существующих DS-компонентов.

## 2. Если похожий компонент уже есть

- **Не верстать заново.**
- Проверить, можно ли адаптировать под MIDHUB (tokens, web-layout, variants).
- Если адаптировать нельзя — **письменно объяснить почему** (конфликт токенов,
  mobile-only логика, layered gaming-визуал Ubuntu и т.п.) и только тогда делать новый.

## 3. Если компонента нет

1. Создать новый компонент в **MIDHUB DS** (`src/components/ds/`).
2. Добавить `variants` / `states` (через `cva` или эквивалент).
3. Добавить пример использования (story / preview / комментарий).
4. **Только потом** использовать его на экране.

## 4. Схема построения экрана

1. Проверить Figma / reference.
2. Найти нужные DS-компоненты.
3. Проверить tokens (цвета, типографика, spacing, radius).
4. **Только потом** верстать.
5. После вёрстки — сравнить результат с Figma / reference.

## 5. Никаких одноразовых локальных компонентов

Если элемент — это **UI-pattern** (кнопка, инпут, карточка, модалка, таб, бейдж),
он живёт в DS, а не локально внутри экрана. Локально — только композиция
DS-компонентов и уникальный одноразовый layout.

## 6. Никакого прямого копирования стилей

Запрещено копировать стили/токены из Ubuntu или Vector «как есть».
Любой перенос — через **адаптацию под MIDHUB tokens**. Хардкод значений
(`#hex`, `px` мимо токенов) запрещён.

## 7. Web-first

- Базовый layout проектируется под **desktop web**.
- Responsive (tablet / mobile) — **отдельным слоем**, через breakpoints, а не наоборот.
- Не тащить Telegram/mobile-специфику (safe-area, TG WebApp API, bottom-tab-bar
  как основная навигация) в базовые решения.

## 8. Не трогать Vector и Ubuntu без разрешения

Файлы `src/` (Vector), `ubuntu-app/`, `Ubuntu Design System/` — **read-only
референс**. Любое изменение в них — только после явного разрешения пользователя.

## 9. Композитные компоненты — тоже в DS

Когда на основе DS собирается **фича = композитный компонент** (несколько
DS-компонентов, объединённых в переиспользуемый блок: карточка урока, форма
логина, шапка профиля, фильтр-панель и т.п.) — этот композитный компонент
**тоже добавляется в DS**, а не остаётся локально внутри экрана.

**Композит в большинстве своём состоит из готовых ui-kit / DS-компонентов.**
Поэтому перед версткой композита нужно **сверить все его части с DS** (см. п. 1):
почти каждый кусок (кнопка, инпут, таб, бейдж, тег, аватар, иконка, тултип
и т.п.) уже есть в `src/components/ds/`. Сначала собрать композит из
существующих компонентов и только потом — если чего-то реально нет — добавлять
новое. Не верстать с нуля то, что уже есть в DS.

Правила:

1. Базовые DS-компоненты живут в `src/components/ds/`. Композитные —
   в `src/components/ds/composite/` (создать папку при необходимости).
2. Композит собирается **только из существующих DS-компонентов и tokens**.
   Если внутри нужен новый базовый UI-элемент — сначала добавить его в DS
   (см. п. 3), потом собирать композит.
3. У композита — свои `props` / `variants` / `states` и пример использования
   (story / preview / комментарий), как у базового компонента.
4. Экран собирается из композитов и базовых DS-компонентов. Локально на экране
   остаётся **только уникальный одноразовый layout** — то, что не переиспользуется.

Критерий «в DS или локально»: переиспользуется хотя бы потенциально или
повторяет паттерн → **в DS**. Строго одноразовая компоновка под конкретный
экран → локально.

## 10. Запуск голосования → активный вопрос + оранжевая иконка в меню

Когда по любой сущности **запускается голосование** (кнопка «Запустить
голосование»: формирование совета, распределение % по счетам и т.п.), состояние
должно отражаться единообразно:

1. **Вопрос подсвечивается в списке «Вопросы голосования»** — оранжевая рамка
   (`--color-orange-500`) + статус «На голосовании», и встаёт **первой строкой**.
   Все источники вопросов (распределение %, этапы совета, повестка) собираются в
   один список, и **активные всегда сортируются наверх** — независимо от источника.
   Завершённые вопросы остаются в списке без подсветки (режим «только результаты»).
2. **Иконка «Вопросы голосования» в боковом меню — оранжевая**
   (`--color-orange-500`), как у прошлых активных вопросов. Оранжевой остаётся,
   **пока есть хотя бы один активный вопрос**; когда активных нет — нейтральная
   (`--color-grey-300`); на самой странице вопросов — синяя (`text-primary`).
3. Источник активности — **общее состояние во `RegFlow`** (флаги вроде
   `votingStarted` / `accountsVoteStarted`), а не локальный стейт экрана: меню и
   список вопросов читают его сами (см. [[midhub-shared-chrome-reads-global-state]]).
4. **Завершение голосования** снимает подсветку/оранжевый и применяет результат
   (напр. проценты распределения → видны на странице счетов).
5. **Голосование считается активным только по факту запуска**, а не пока сущность
   ещё формируется. Пока кандидат не выбран / приглашения не отправлены —
   вопрос **не создаётся** в списке, слот не помечается «На голосовании», иконка в
   меню **не оранжевая**. Нельзя завязывать активность на «этап ещё не завершён»
   (`councilStage < 3` и т.п.) или на простой факт нахождения на экране вопросов —
   только на реальные флаги запуска (`votingStarted` / `accountsVoteStarted`).

Реализация-референс: распределение % целевого счёта — `AccountSettingsScreen`
(«Запустить голосование» → `startAccountsVote`) → вопрос «Распределение % по
подсчетам целевого счета» в `VotingQuestionsScreen` → `AccountVotingScreen` →
`finishAccountsVote` (проценты на `AccountsScreen`).
