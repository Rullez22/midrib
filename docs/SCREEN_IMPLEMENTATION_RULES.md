# MIDHUB — Правила реализации экранов

> Эти правила обязательны при реализации **любого** нового экрана MIDHUB.
> Связанные документы: [DS_INVENTORY.md](./DS_INVENTORY.md), [MIDHUB_WORKFLOW_RULES.md](./MIDHUB_WORKFLOW_RULES.md).

---

## Основной принцип

MIDHUB уже имеет полноценную Design System.

Большинство будущих экранов будут состоять из уже существующих компонентов.

Поэтому **запрещено** создавать новые компоненты, стили, токены или layout-решения без предварительного анализа существующей DS.

DS является **источником истины**. Экран не должен создавать собственную дизайн-систему внутри себя.

---

## Обязательный процесс перед реализацией любого экрана

Каждый раз перед началом работы над новым экраном выполняй следующий алгоритм.

### Шаг 1. Анализ экрана

Изучить экран целиком. Разбить его на:

- layout
- sections
- blocks
- components
- states
- interactions

Составить внутреннюю карту экрана.

### Шаг 2. Поиск совпадений в DS

Для каждого элемента экрана проверить:

- tokens
- typography
- colors
- spacing
- icons
- buttons
- inputs
- cards
- tabs
- modals
- dropdowns
- navigation
- list items
- containers
- overlays
- sheets
- badges
- avatars
- status indicators
- loaders
- empty states

Проверить:

- есть ли готовый компонент
- есть ли похожий компонент
- можно ли расширить существующий variant

### Шаг 3. Переиспользование

Приоритет всегда такой:

1. Использовать существующий компонент **без изменений**.
2. Использовать существующий компонент через **variant**.
3. **Расширить** существующий компонент.
4. Только если ничего не подходит — **создать новый** DS-компонент.

Создание нового компонента — крайний случай.

---

## Запрещено

Запрещено дублировать, если аналог уже существует:

- Button
- Card
- Modal
- Input
- ListItem
- Tab
- Navigation
- Overlay

---

## Проверка перед созданием нового компонента

Перед созданием нового компонента обязательно ответить на вопросы:

1. Есть ли уже такой компонент?
2. Есть ли похожий компонент?
3. Можно ли добавить новый variant?
4. Можно ли собрать решение из существующих компонентов?

**Если ответ «да» хотя бы на один пункт — новый компонент создавать нельзя.**

---

## Работа с экранами

При реализации нового экрана:

1. Сначала собрать экран из существующих DS-компонентов.
2. Только после этого определить:
   - какие части отсутствуют
   - какие variants нужно добавить
   - какие новые компоненты действительно необходимы

---

## Работа с токенами

Запрещено использовать:

- случайные цвета
- случайные размеры
- случайные радиусы
- случайные отступы
- локальные box-shadow

Использовать только **DS tokens**.

Если нужного токена нет — сначала предложить добавить его в DS.

---

## Работа со стилями

Запрещено:

- копировать стили прямо внутрь экрана
- создавать screen-specific button
- создавать screen-specific card
- создавать screen-specific modal

Если стиль нужен более одного раза — он должен попасть в DS.

---

## Отчёт перед реализацией каждого экрана

Перед началом реализации выводить блок **Reuse Analysis**:

```
## Reuse Analysis

Найдено компонентов DS:
- ...

Будут переиспользованы:
- ...

Нужны новые variants:
- ...

Нужны новые компоненты:
- ...

Причина создания новых компонентов:
- ...
```

Только после этого приступать к реализации.

---

## Цель

Добиться того, чтобы **минимум 80–90%** каждого нового экрана собиралось из уже существующих DS-компонентов.

DS является источником истины. Экран не должен создавать собственную дизайн-систему внутри себя.
