# MIDHUB — Responsive First Rules

> Эти правила обязательны при реализации **любого** нового экрана MIDHUB.
> Связанные документы: [SCREEN_IMPLEMENTATION_RULES.md](./SCREEN_IMPLEMENTATION_RULES.md), [DS_INVENTORY.md](./DS_INVENTORY.md), [MIDHUB_WORKFLOW_RULES.md](./MIDHUB_WORKFLOW_RULES.md).

---

## Важное правило

Ссылки на экраны приходят из Figma. Во многих случаях Figma содержит только **WEB-версию** экрана.

Это **НЕ** означает, что нужно реализовать только WEB.

Каждый экран автоматически проектируется как **responsive screen**.

---

## Обязательные платформы

Каждый новый экран должен поддерживать **без дополнительных указаний**:

- Mobile
- Tablet
- Desktop
- Large Desktop

---

## Responsive First подход

Запрещено:

- делать экран только под Desktop
- делать фиксированные ширины без необходимости
- делать layout, который ломается на планшете
- делать layout, который ломается на мобильных устройствах

Каждый экран сразу проектируется как адаптивный.

---

## Перед вёрсткой экрана — анализ по breakpoints

### Mobile — 320–767 px

Проверить: навигацию, карточки, списки, таблицы, модалки, формы.

### Tablet — 768–1279 px

Проверить: количество колонок, размеры контейнеров, отступы, перестроение layout.

### Desktop — 1280–1919 px

Использовать дизайн из Figma как основной референс.

### Large Desktop — 1920+ px

Проверить: максимальную ширину контента, контейнеры, растяжение секций, большие пустоты.

---

## Если в Figma есть только WEB

Если мобильной версии нет — **не придумывать дизайн заново**. Нужно:

1. Взять Desktop экран.
2. Сохранить визуальную систему.
3. Адаптировать layout под mobile и tablet.
4. Использовать существующие компоненты DS.
5. Следовать общим паттернам MIDHUB.

---

## Обязательная проверка перед завершением задачи

### Mobile

- нет горизонтального скролла
- все элементы доступны
- все кнопки кликабельны

### Tablet

- layout не ломается
- контент не обрезается

### Desktop

- соответствует Figma

### Large Desktop

- нет растянутых блоков
- контент остаётся читаемым

---

## Работа с компонентами DS

Все компоненты DS должны сразу поддерживать mobile / tablet / desktop.

**Запрещено** создавать компонент только под один breakpoint.

---

## Если создаётся новый компонент

Перед реализацией определить:

- mobile variant
- tablet variant
- desktop variant

Если компонент не адаптивен — он считается **незавершённым**.

---

## Отчёт перед реализацией экрана

Перед началом работы выводить блок **Responsive Analysis**:

```
## Responsive Analysis

Screen:
...

Target breakpoints:
- Mobile
- Tablet
- Desktop
- Large Desktop

Layout changes:
- ...

Components reused:
- ...

New variants required:
- ...

Potential responsive risks:
- ...
```

Только после этого приступать к реализации.

---

## Основная цель

Каждый экран MIDHUB должен быть **production-ready одновременно** для Mobile, Tablet и Desktop — даже если в Figma присутствует только Desktop-версия экрана.
