# Флоу создания договора у партнёра (Договор → подпись → консультант)

Дата: 2026-07-02. Экран: `/cabinet/[company]/partners/org/[id]` (таб «Документооборот»)
и окно документа `/cabinet/[company]/partners/org/[id]/doc/[docId]`.

Источник — Figma «Midhub ERP» (file 8QfpCJU0RUURop4eh5qZ5O). Работает во всех
кабинетах и у всех партнёров/организаций.

## Машина состояний

1. **Документооборот** → кнопка «Добавить документ» открывает **дропдаун**
   (6760-467314): `Договор · Счёт на оплату · Акт выполненных работ`. Реализуем
   только **Договор** (остальные пункты убраны по решению пользователя).
2. **Форма «Добавление договора»** (6760-466719 пустая / 6760-466856 заполненная):
   Название\*, Номер\*, Код\* (?), Сумма, Комментарии + upload-зона → прикреплённый
   документ (Изменить/Удалить). Кнопки: «Добавить договор» (активна при валидности),
   «Отменить» (появляется когда приложен документ) → назад.
3. «Добавить договор» → **попап подтверждения в блокчейн** (6760-505279, ре-юз DS Modal).
4. Подтверждаем → **созданный договор** (6760-467754): данные из формы, чат пустой,
   Документы пустые, блокчейн `Добавление договора`, кнопки «Редактировать»(disabled)
   + «Подписать»(active). Tabs Договор/Консультант.
5. Назад → в списке строка со статусом **«Ожидает участия»** (оранжевая рамка, есть).

### Развилка «Подпись»
- «Подписать» → DS Modal → **ожидание** (6760-468038): «Подписать» disabled, блокчейн
  += `Подпись исполнителя`, чат с сообщениями. Статус в списке ещё «Ожидает участия».
- **Через 5 сек** (таймер в провайдере) → **финал** (6760-469868): единый блок
  «Согласован» (уже собран в org-contract-screen). Статус в списке → «Согласован».

### Ветка «Консультант» (таб Консультант, до или после подписи)
- Поиск (6760-471481): живой фильтр по названию (пусто → карточка скрыта).
  Карточка: фото, имя, 5★, услуги, «Выбрать консультанта» + «Подробнее».
- «Выбрать» (6760-471797): поиск скрывается, у карточки «Отменить выбор»; ниже
  «ШАГ 2 из 2»: Сумма\* + «Доступная сумма: 250 000 ₽», Описание задачи\*.
- Заполнили (6760-472053): «Добавить консультанта» active + «Отменить»(→ назад к поиску).
- «Добавить консультанта» → **окно консультации** (6760-472889): плашка «Ваш консультант»
  (сворачиваемая) + Документы/Публикация (ре-юз DocsTable/FeedComposerBar) + «Чат с
  консультантом» + «Процесс исполнения консультации». Кнопка «Завершить консультацию».
- «Завершить» → окно документа: блокчейн += `Добавление консультанта`; вид зависит
  от того, добавляли до/после подписи (6760-467754-подобный active-Подписать, либо
  6760-468356 — обе кнопки disabled + все 3 tx).

## Данные / состояние (RegFlow, in-memory, переживает клиентскую навигацию)
- `createdContracts: CreatedContract[]` (из формы; поля Название/Номер/Код/Сумма/
  Комментарии/attached + orgId + customer/executor + signed/finalized + consultant).
- методы: `addCreatedContract`, `getCreatedContract`, `signCreatedContract` (+5с→finalize),
  `finalizeContract`, `addConsultant`, `finishConsultation`.

## Ре-юз
OrgContractScreen (подписанный/неподписанный вид, tabs, модалка, чат, блокчейн,
DocsTable, FeedComposerBar), строки «Ожидает участия» в DocFlow, DS Dropdown/Modal/
Input/Textarea/Button/Badge/UploadV2/ChatPanel/BlockchainCard.

## Расширение (2026-07-02, вторая часть): Счёт на оплату + Акт выполненных работ

Флоу договора обобщён в параметризованный по виду документа (`DocKind` =
contract/invoice/act, конфиг в `doc-kinds.ts`). Добавлены:
- **Счет на оплату** и **Акт выполненных работ** в дропдаун «Добавить документ»
  партнёра и в дропдаун внутри документа. Форма/попап/created/подпись/консультант —
  те же, метки из `KINDS[kind]`.
- **Авто-прогресс после подписи**: транзакции + сообщения чата появляются каждые 2с
  (счёт — 3 шага, акт — 2 шага), тексты 1:1 из Figma; затем финализация (счёт +2с,
  акт +3с). Таймер в RegFlow (`KIND_PROGRESS`, поле `step`).
- **Акт — терминальный**: у готового акта нет блока «Документы», только «Публикация».
- **Редактировать** активна в состоянии «создан, не подписан» → форма с
  предзаполненными данными (`?edit=<id>`, `updateCreatedContract`).
- **Вложенность** (`parentId`): документ, созданный внутри блока «Документы» другого
  документа, — вложенный (`NestedDocsBlock`, таблица Тип/Сумма/Статус/Дата, Figma
  503641); созданный на странице партнёра — верхнеуровневый. Проброс статуса:
  родитель оранжевый + второй бейдж «Ожидает участия», если вложенный требует действий.

Файлы: `reg-flow.tsx`, `doc-kinds.ts` (нов.), `contract-create-screen.tsx`,
`created-contract-screen.tsx`, `nested-docs-block.tsx` (нов.), `partner-doc-screen.tsx`,
`org-detail-screen.tsx`, `contract-new/page.tsx`, `doc/[docId]/page.tsx`.

Проверено Playwright: счёт (created→подпись→3 авто-tx/чата→Согласован), акт
(терминальный, 4 tx, только Публикация), вложенность (вложенный акт в таблице
родителя, в списке партнёра его нет, проброс «Ожидает участия»). Typecheck чистый.

## Исключение: «Оценка и закрытие договора» (culture / Договор №1)

Партнёр «Живу с Культурой», Договор №1 (Ожидает участия) — особый флоу
(Figma 6760-501003 … 501941), `culture-doc-screen.tsx`:
- Открытие → статус **«Оценка»** (оранжевый, и на плашке в списке партнёра).
- Экран «Оценка и закрытие договора»: 2 карточки участников (Фонд Живу С Культурой,
  ООО «MIDHUB») — лого, услуги, «Подробнее», **кликабельные звёзды** (1–5),
  текст отзыва, «Оставить отзыв» → попап (DS Modal, подтверждение в блокчейн).
- Как только оставлены **2 отзыва** → через **3с** статус **«Закрыт»** (серый):
  инфо-баннер «Обратите внимание…», таблица готовых документов (акты/счета),
  Публикация с лентой; полей/кнопок добавления договора больше нет.
- Статусы в RegFlow: `cultureStarted/cultureReviews/cultureClosed`. Роутинг —
  `doc/[docId]/page.tsx` (org=culture && Договор №1 → CultureDocScreen).

Проверено Playwright: Оценка (2 карточки, звёзды, попап) → 2 отзыва → «Закрыт»
(баннер + документы) → плашка в списке «Закрыт». Typecheck чистый.

## Итог
Реализован целиком. Весь флоу прогнан end-to-end (Playwright): форма→попап→created→
«Ожидает участия»→подпись→ожидание(Подписать disabled)→+5с финал «Согласован»;
ветка консультанта: поиск→выбор→форма→окно консультации→«Завершить»→tx «Добавление
консультанта». Экраны сверены со скриншотами Figma. typecheck чистый.

Файлы: `reg-flow.tsx` (состояние), `contract-create-screen.tsx` (+роут `contract-new`),
`created-contract-screen.tsx` (окно договора/консультанта), `org-detail-screen.tsx`
(дропдаун + строки), `doc/[docId]/page.tsx` (роутинг velesta↔created),
`org-contract-screen.tsx` (экспорт переиспользуемых хелперов).

Ограничение (как и везде в приложении): состояние in-memory — переживает клиентскую
навигацию, но не полную перезагрузку страницы (тогда created-договор показывает
fallback «Договор не найден»).
